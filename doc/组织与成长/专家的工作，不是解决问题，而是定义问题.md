# 专家的工作，不是解决问题，而是定义问题

副标题：为什么越往上走，越像产品经理

最近一段时间，我发现自己的工作发生了一种很奇怪的变化。以前一天结束的时候，我总能很轻松地回答一个问题：今天做了什么。因为答案通常非常具体，写了多少代码，修了多少 Bug，上线了什么功能，完成了哪个项目。哪怕什么都不看，只要翻一下 Git 提交记录，基本就能知道这一天有没有价值。但现在越来越难回答这个问题了。有时候开了一天会，有时候讨论了一天方向，有时候和不同团队反复沟通，有时候为了一个能力的定位争论几个小时。晚上回家的时候，经常会产生一种奇怪的感觉：如果用工程师的标准来看，好像什么都没干，没有写代码，没有上线功能，也没有解决具体 Bug，但又隐约觉得，这些事情似乎比写代码更重要。

这种感觉持续了很长时间。最开始我以为这只是职级提升之后必然出现的“杂事变多”，后来才慢慢意识到，也许问题并不在于我离技术越来越远了，而在于我面对的问题已经变了。以前的问题大多是明确的，登录失败了，接口慢了，数据库扛不住了，系统崩了，页面白屏了。大家都知道问题是什么，区别只是看谁能更快找到原因，谁能设计出更好的方案，谁能把问题解决得更彻底。所以在工程师阶段，优秀与否往往很容易判断：给你一个问题，你能不能解决。

这几乎构成了大部分工程师职业生涯前半段的核心训练。学校教我们解题，工作教我们解决问题，面试考察我们分析问题，技术社区奖励我们解决问题。久而久之，我们会形成一种非常自然的认知：技术人的价值来自解决问题的能力。这套逻辑在工程师阶段几乎完全正确，但后来我发现，它并不能解释专家阶段发生的很多事情。

有一次讨论一个新方向的时候，会议开了很久。有人说应该建设统一平台，有人说应该补齐底层能力，有人说应该沉淀知识体系，也有人说应该做自动化能力。大家讨论得非常激烈，但越讨论，我越觉得不对劲。因为所有人都在讨论答案，却没有人讨论问题。这些看起来不同的方案，也许根本不是竞争关系，它们只是不同人眼中的答案。而大家真正缺失的，是对问题本身的共识：到底是什么问题，值得我们投入一年时间去解决？如果这个问题没有想清楚，那么后面所有关于方案的讨论，本质上都只是空转。

后来我越来越觉得，很多组织里的低效，并不是因为不会解决问题，而是因为从一开始就定义错了问题。大部分人会认为组织最大的浪费来自执行，比如需求做错了、项目延期了、代码质量差了、资源投入错了。但这些其实都是后果。真正昂贵的成本，往往发生在更早的阶段：问题定义错了。

比如很多团队做数字化转型的时候，会把问题定义成“我们缺少数字化系统”，于是开始建设系统。几年过去，系统越来越多，效率却没有明显提升。后来才发现，真正的问题不是缺系统，而是流程本身没有标准化，责任边界没有理清，数据口径也没有统一。系统只是把原来的混乱数字化了一遍。再比如很多团队讨论 AI 的时候，会很快得出一个答案：我们需要一个 AI Agent。但真正落地之后才发现，效果并不取决于 Agent 本身，而取决于知识是否结构化、流程是否标准化、数据是否可用、权限和责任是否清晰。Agent 只是把原来被隐藏的问题暴露得更明显。

这些案例有一个共同点：组织不是败给了解决方案，而是败给了问题定义。因为错误的问题，会产生正确不了的答案。一个错误的问题越早被转化为项目，后面的执行越高效，浪费反而越大。因为所有人都在用非常专业的能力，解决一个一开始就不值得这样解决的问题。

也是在这个阶段，我开始重新理解工程师和专家之间的区别。过去我会觉得专家只是更厉害的工程师，能解决更复杂的问题，能设计更大的系统，能做更困难的技术决策。但现在我越来越觉得，专家和工程师最大的区别，并不只是解决问题的能力差异，而是两个人面对的问题已经不是同一个层级的问题。工程师解决的是已经被定义好的问题，而专家更重要的工作，是在一堆混乱的现象里定义出真正的问题。

解决问题的时候，你面对的是一个已经被切好的问题，边界是明确的，目标是明确的，成功标准也是明确的。你需要在约束条件下找到一个最优解，这是一种局部优化能力。但定义问题的时候，你面对的是混乱。不同的人有不同观点，不同团队有不同诉求，甚至很多时候，大家连问题本身都没有共识。你看到的不是问题，而是一堆现象、一堆抱怨、一堆需求、一堆方案。你的工作是从这些东西里找到真正的问题，并让足够多的人相信这才是值得解决的问题。

这也是为什么很多技术专家越往后越像产品经理。以前我对此一直有点困惑：为什么一个技术专家每天都在讨论用户？为什么一个架构师天天在讲业务？为什么一个负责人总是在问“为什么做”，而不是直接讨论“怎么做”？后来我才发现，随着影响范围变大，技术人面对的问题开始从 How 变成 What 和 Why。工程师更关注“怎么做”，专家必须先判断“做什么”以及“为什么做”。如果 What 错了，How 再漂亮也没有意义；如果 Why 错了，整个项目可能从一开始就在错误方向上加速。

所以很多时候，专家最重要的工作并不是设计答案，而是找到真正的问题。这个过程不像写代码那样有即时反馈，也不像修 Bug 那样很快能看到结果。它经常是模糊的、缓慢的、充满争论的。你需要不断追问：这个需求背后到底是什么问题？这个方案是在解决问题，还是在制造新的复杂度？这个问题如果不解决，会发生什么？这个问题是不是现在最值得解决的问题？

我后来慢慢接受了一个现实：在组织里，答案其实没有想象中稀缺，真正稀缺的是问题。因为一个好问题会自动吸引很多好答案，而一个坏问题会消耗所有人的努力。以前我觉得专家是最厉害的工程师，后来我觉得专家是最会做技术决策的人，现在我越来越觉得，专家首先应该是问题定义者。因为一个组织里最稀缺的资源，不是技术，不是人力，甚至不是预算，而是注意力。每一个问题定义，都在决定组织把注意力投向哪里。而注意力最终决定资源流向，资源流向决定行动，行动决定结果。从这个意义上来说，问题定义本身，就是一种资源配置行为。

回过头来看，为什么我越来越难回答“今天做了什么”这个问题？因为工作的重心已经变了。以前我的价值来自解决问题，现在越来越多的价值来自发现问题、定义问题，以及推动问题被解决。这些事情不像写代码那样有即时反馈，它们看起来很虚，甚至很长时间都看不到结果。但它们决定了后面无数人的工作方向。

很多技术人的成长路径可以被描述成：写代码、解决问题、设计系统、制定方向。但我觉得更准确的描述应该是：解决问题、发现问题、定义问题、推动问题被解决。这是一个能力维度的升级，也是很多技术人成长过程中最容易忽略的一步。因为解决问题会带来即时反馈，而定义问题往往需要很长时间才能看到结果。但真正决定一个组织上限的，往往不是谁最会解决问题，而是谁最先看到了真正的问题。

所以我现在越来越相信一句话：工程师创造答案，专家定义问题。而一个组织能够走多远，很多时候取决于它是否问对了问题。
