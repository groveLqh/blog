# Agent 面试知识点 #007：Agent 为什么会陷入死循环？

## 摘要

Agent 死循环是 Agent 系统里非常常见的问题。

表面上看，是模型一直在重复思考、重复调用工具、重复尝试同一个动作。

但本质上不是模型“笨”，而是系统没有告诉它：

```text
什么时候算完成？
什么时候应该停止？
什么时候应该换路径？
什么时候应该请求人介入？
哪些动作已经失败过，不能继续重复？
```

所以 Agent 死循环不是单纯的 Prompt 问题，而是 Agent Runtime 的控制问题。

一个成熟的 Agent 系统，必须用 State、停止条件、重试策略、工具调用记录、执行预算和人工确认机制，来防止 Agent 在不确定任务里无限循环。

---

## 问题

Agent 为什么会陷入死循环？

如何在真实系统里避免 Agent 一直重复思考、重复调用工具、重复执行同一个动作？

---

## 面试官在考什么？

这个问题表面是在问 Agent 的异常行为，实际上是在考你是否理解 Agent 和普通 LLM 调用的区别。

普通 LLM 调用通常是：

```text
输入 → 输出 → 结束
```

但 Agent 是：

```text
目标 → 计划 → 执行 → 观察 → 更新状态 → 再决策
```

只要进入“再决策”，就一定会出现循环结构。

循环本身不是问题。

Agent 必须循环，才能完成多步任务。

真正的问题是：

```text
Agent 不知道循环什么时候应该结束。
```

比如一个数字员工要帮用户处理工单。

它可能会：

```text
读取工单 → 查询订单 → 判断问题 → 调用售后系统 → 生成回复 → 等待确认
```

但如果工具返回为空、权限不足、页面没加载出来、子 Agent 没有返回结果，Agent 就可能不断重复：

```text
再查一次
再试一次
重新分析一下
重新调用工具
重新生成计划
```

如果 Runtime 没有控制，它就会陷入死循环。

所以面试官真正想听到的不是“加一个最大轮数”，而是：

> Agent 死循环的本质，是目标、状态、工具反馈、失败记录和停止条件没有被 Runtime 正确管理。

---

## 核心知识点

### 1. Agent 死循环通常来自目标不收敛

很多 Agent 任务一开始就没有清晰的完成标准。

比如：

```text
帮我分析这个合同
帮我优化这个页面
帮我处理这个客户问题
帮我写一份方案
```

这些目标都很开放。

如果没有定义完成条件，Agent 就可能一直觉得“还可以再优化一点”。

所以 Runtime 需要把目标拆成可结束的状态：

```text
合同风险点已识别
修改建议已生成
用户已确认
结果已写入文档
任务完成
```

否则 Agent 会不断在“继续分析”和“重新总结”之间打转。

---

### 2. State 不完整会导致重复执行

Agent 每一步都依赖 State。

如果 Runtime 没有记录已经发生过什么，Agent 就会像失忆一样重复操作。

比如 MCP 工具调用里，Agent 已经查过一次订单状态：

```text
工具：query_order_status
参数：order_id=12345
结果：未发货
```

如果这个结果没有进入 State，下一轮模型可能又会调用同一个工具，甚至用同样的参数调用多次。

这不是工具的问题，而是状态管理的问题。

State 至少要记录：

```text
已执行动作
工具调用参数
工具返回结果
失败次数
当前阶段
下一步候选动作
是否需要用户确认
```

没有 State，Agent 就无法判断“我是不是已经做过这件事”。

---

### 3. 工具失败没有退出策略

很多死循环发生在工具失败之后。

比如：

```text
文件读取失败
页面元素找不到
接口超时
MCP 工具返回空
ACP 操作没有生效
子 Agent 没有返回
```

如果 Agent 只知道“失败了要重试”，但不知道“最多重试几次”“什么时候换工具”“什么时候停止”，就会无限重试。

正确做法是给每类失败设置策略：

```text
可重试失败：最多重试 2 次
参数错误：重新生成参数
权限不足：请求用户授权
页面不可见：切换到 ACP / 浏览器控制
连续失败：停止并报告原因
```

Agent 不能只有执行能力，还要有失败处理能力。

---

### 4. Reflection 也可能制造死循环

Reflection 的作用是让 Agent 反思自己的结果。

但如果没有边界，Reflection 本身也会循环。

比如：

```text
生成答案 → 反思不够好 → 修改答案 → 再反思 → 再修改 → 再反思
```

最后 Agent 会一直觉得结果还能更好。

所以 Reflection 必须有退出条件：

```text
最多反思 N 次
只检查关键风险
只在失败时触发
评分达到阈值即停止
发现无法改进则结束
```

Reflection 不是让 Agent 无限自我怀疑，而是让 Agent 在有限成本内做质量校验。

---

### 5. 多 Agent 协作会放大死循环

A2A 场景下，死循环更隐蔽。

主 Agent 可能等待子 Agent，子 Agent 又等待另一个 Agent。

比如：

```text
主 Agent：等待风险分析 Agent 返回
风险分析 Agent：等待案例检索 Agent 返回
案例检索 Agent：请求主 Agent 补充上下文
主 Agent：继续等待风险分析 Agent
```

这就是多 Agent 里的循环等待。

所以 A2A 协作必须有明确的任务状态：

```text
pending
running
blocked
completed
failed
timeout
needs_human
```

主 Agent 不能只知道“任务发出去了”，还要知道子任务卡在哪里。

否则多 Agent 不会更聪明，只会更混乱。

## 工业界方案

### 1. Runtime 用状态机管理任务阶段

成熟的 Agent Runtime 不会让模型自由循环，而是用状态机控制任务推进。

例如：

```text
created
→ planning
→ executing
→ verifying
→ waiting_user_confirm
→ completed
```

异常路径则是：

```text
executing
→ failed
→ retrying
→ replanning
→ needs_human
→ stopped
```

每个状态都规定：

```text
允许执行什么动作
禁止执行什么动作
最多执行几次
什么条件进入下一步
什么条件终止任务
```

这样 Agent 不是“想做什么就做什么”，而是在 Runtime 约束下执行任务。

---

### 2. MCP 工具调用要做去重和预算控制

在 MCP 场景里，Agent 会频繁调用外部工具。

Runtime 应该记录每次工具调用：

```text
tool_name
arguments
result
timestamp
status
retry_count
side_effect
```

然后做控制：

```text
相同参数的只读工具调用可以复用结果
写操作必须要求用户确认
同一工具连续失败后停止重试
高成本工具调用需要预算限制
```

尤其是写操作，不能让 Agent 在死循环中重复提交、重复发消息、重复改数据。

---

### 3. ACP 场景要记录界面操作结果

ACP 偏向 Agent 操作 IDE、桌面应用、浏览器和本地环境。

这类场景死循环很常见。

比如 Agent 一直点击按钮，但页面没有变化。

Runtime 必须记录：

```text
当前窗口
当前页面
上一次操作
操作前截图 / DOM / 状态
操作后结果
是否发生变化
失败次数
```

如果连续几次操作后环境没有变化，就应该停止，而不是继续点击。

桌面数字员工尤其需要这个能力。

因为它操作的是真实系统，不是模拟环境。

---

### 4. A2A 协作要有超时和阻塞状态

多 Agent 系统里，任务不能只靠自然语言传递。

每个子任务都应该有结构化状态：

```text
任务目标
输入上下文
期望输出
当前状态
截止时间
失败原因
返回结果
置信度
```

如果子 Agent 超时，主 Agent 应该能选择：

```text
重新分派
降级处理
请求人工介入
跳过该子任务
结束任务
```

否则多个 Agent 之间会互相等待，形成系统级死循环。

---

### 5. 数字员工必须有人工接管点

数字员工不是聊天机器人，它会连接企业系统、审批流、IM、浏览器、文件和业务后台。

所以数字员工不能无限执行。

它必须在关键节点停下来：

```text
权限不足
高风险写操作
连续失败
结果不确定
涉及金额 / 合同 / 客户通知
需要业务判断
```

这类情况不应该让 Agent 硬做，而应该进入：

```text
needs_human
```

也就是请求人工确认。

一个好的数字员工，不是永远自动化，而是知道什么时候该停下来。

## 常见误区

### 误区一：以为加 max_steps 就够了

最大步数只能防止系统无限跑下去，但不能解决为什么会死循环。

真正要解决的是：

```text
为什么没有完成条件？
为什么重复调用同一个工具？
为什么失败后还在重试？
为什么没有进入人工确认？
```

max_steps 是保险丝，不是治理方案。

---

### 误区二：把死循环归因于模型能力差

模型能力差确实会增加循环概率，但不是根因。

很多时候，即使用更强的模型，如果 Runtime 没有状态管理和停止条件，仍然会死循环。

Agent 工程不能只靠换模型。

---

### 误区三：只记录成功，不记录失败

失败记录非常重要。

如果 Agent 不知道自己失败过，就会重复失败。

State 里应该记录：

```text
失败动作
失败原因
失败次数
是否已经换过路径
是否已经请求过人工介入
```

失败不是垃圾信息，而是下一步决策的重要依据。

---

### 误区四：让 Agent 自己决定所有停止条件

完全让模型自己判断“要不要继续”，风险很高。

模型可能会过度努力，也可能过早结束。

工业系统里，停止条件应该由 Runtime 和业务规则共同控制。

比如：

```text
工具失败超过 3 次就停止
写操作必须等用户确认
子任务超时后降级
成本超过预算后终止
质量评分达标后结束
```

这些规则不能只写在 Prompt 里。

---

## 面试加分答案

如果面试官问：

> Agent 为什么会陷入死循环？

可以这样回答：

```text
Agent 会陷入死循环，本质上是因为它处在一个“计划—执行—观察—再决策”的循环结构里，但系统没有给它明确的完成条件、失败退出策略、状态记录和执行预算。

我会从四个层面理解这个问题：

第一，目标不收敛。比如“继续优化”“继续分析”这类任务，如果没有完成标准，Agent 会一直尝试改进。

第二，State 不完整。Agent 如果不知道自己已经调用过哪些工具、哪些参数失败过、当前任务处于哪个阶段，就会重复执行同样动作。

第三，工具调用没有治理。MCP 工具返回空、ACP 页面操作无变化、接口失败或权限不足时，如果没有重试上限、降级策略和人工确认机制，就会反复调用。

第四，多 Agent 协作会放大循环。A2A 场景下，如果主 Agent 和子 Agent 没有结构化任务状态、超时机制和阻塞状态，就可能互相等待。

所以解决方案不是只加 max_steps，而是由 Agent Runtime 做系统性控制：用状态机管理任务阶段，用 State 记录执行历史，用预算和重试策略限制循环，用工具调用去重避免重复动作，用 needs_human 状态处理不确定和高风险场景。

简单说，Agent 死循环不是一个 Prompt 问题，而是 Runtime 控制问题。
```

这个回答的加分点在于：

你没有把死循环归因于“模型不够聪明”，而是把它放到了 Agent Runtime、状态管理、工具治理和多 Agent 协作的系统设计里。

---

## 一句话总结

Agent 会死循环，是因为它能持续执行，却没有被 Runtime 明确告知什么时候完成、什么时候重试、什么时候换路径、什么时候停止、什么时候请求人工介入。
