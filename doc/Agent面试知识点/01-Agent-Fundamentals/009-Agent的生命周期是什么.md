# Agent 面试知识点 #009：Agent 的生命周期是什么？

## 摘要

很多人理解 Agent，容易把它理解成：

```text
用户输入一句话
模型思考一下
调用几个工具
最后返回答案
```

但真正的 Agent 系统不是一次对话，也不是一次工具调用。

它更像一个持续运行的任务执行单元。

一个 Agent 从接收到目标开始，会经历任务理解、规划、工具选择、执行、观察、状态更新、错误恢复、结果交付等阶段。

所以 Agent 的生命周期，本质上是在回答一个问题：

> 一个 Agent 从“接到任务”到“完成任务”，中间到底经历了什么？

如果把生命周期理解清楚，就能更好地设计 Agent Runtime、Memory、Tool Use、MCP 工具、ACP 操作、A2A 协作，以及数字员工的执行闭环。

---

## 问题

Agent 的生命周期是什么？

在真实系统里，一个 Agent 从接收任务到完成交付，一般会经历哪些阶段？这些阶段分别解决什么问题？

---

## 面试官在考什么？

这个问题表面上是在问一个流程。

但面试官真正想考的是：

> 你是否把 Agent 理解成一个可运行、可恢复、可观测、可治理的系统，而不是一次 LLM 调用。

普通 ChatBot 的生命周期很短：

```text
用户输入
模型生成
返回答案
```

但 Agent 不一样。

Agent 会做计划，会调用工具，会和外部系统交互，会失败，会重试，会等待用户确认，也可能把任务交给其他 Agent。

比如一个数字员工处理客户退款：

```text
读取用户消息
理解退款诉求
查询订单
判断规则
调用退款系统
生成回复
通知客户
记录工单
```

这里每一步都可能失败。

订单系统可能超时，退款工具可能没有权限，用户信息可能缺失，规则可能需要人工确认。

所以 Agent 的生命周期不能只看“模型生成了什么”，而要看：

```text
任务状态如何变化
工具是否真的执行
外部环境是否反馈成功
失败后如何恢复
最终交付是否有证据
```

这就是 Agent Runtime 要解决的问题。

---

## 核心知识点

### 1. 接收目标：Goal Intake

Agent 生命周期的第一步，是接收用户目标。

用户通常不会给出一个标准化任务，而是给出自然语言需求。

比如：

```text
帮我整理今天客户投诉里需要优先处理的问题
```

这句话背后其实包含很多隐含信息：

```text
客户投诉在哪里
什么叫优先
需要输出什么格式
是否要通知别人
是否允许写入系统
```

所以 Agent 首先要做目标理解。

它需要把自然语言目标转成可执行任务。

在 Runtime 里，这一步通常会生成一个任务对象：

```json
{
  "goal": "整理客户投诉中的优先问题",
  "status": "created",
  "constraints": ["不直接回复客户", "只输出建议"],
  "required_context": ["投诉记录", "客户等级", "历史工单"]
}
```

如果目标不清楚，Agent 不应该乱做，而应该补充澄清，或者在安全范围内先做只读分析。

---

### 2. 构建上下文：Context Building

Agent 不是凭空执行任务。

它需要知道当前任务相关的上下文。

上下文可能来自：

```text
用户输入
历史对话
文件
数据库
企业系统
Memory
MCP 工具返回
其他 Agent 的结果
```

比如数字员工要处理退款，就需要订单、支付、物流、售后规则等上下文。

这一步很容易出问题。

如果上下文缺失，Agent 可能会把推测当事实。

所以成熟系统会区分：

```text
已知事实
模型推断
用户偏好
外部系统状态
待确认信息
```

Agent 不能把“我猜测用户想退款”当成“用户已经申请退款”。

这是生命周期里非常关键的一步。

---

### 3. 制定计划：Planning

有了目标和上下文之后，Agent 需要拆解任务。

比如：

```text
目标：处理退款请求

计划：
1. 查询订单
2. 查询物流状态
3. 判断是否符合退款规则
4. 如果符合，生成退款建议
5. 如果涉及金额操作，请用户确认
```

Planning 的作用不是让 Agent 显得更聪明，而是让任务过程可控。

没有计划的 Agent 很容易变成：

```text
想到哪做到哪
调用工具没有顺序
失败后不知道回退
最终结果不可解释
```

计划可以是一次性生成，也可以是动态更新。

在复杂任务里，更常见的是：

```text
先生成粗计划
每一步执行后根据结果调整
```

这就是 ReAct、Plan-and-Execute、Reflection 等模式背后的原因。

---

### 4. 选择工具：Tool Selection

Agent 有计划之后，需要决定调用哪些工具。

工具可能来自：

```text
MCP 工具
本地函数
浏览器操作
企业 API
数据库查询
文件读写
消息发送系统
```

这一阶段最重要的是边界。

Agent 不能看到所有工具，也不能随便使用所有工具。

Runtime 应该根据任务动态暴露工具。

比如处理订单问题时，只暴露：

```text
query_order
query_payment
query_logistics
create_ticket
```

不应该同时暴露：

```text
delete_contract
transfer_money
send_company_wide_email
```

工具选择阶段的核心问题是：

```text
当前任务需要什么工具
这个工具是否允许调用
是否是只读操作
是否有副作用
是否需要用户确认
```

这也是 MCP 工具治理的重要部分。

---

### 5. 执行动作：Execution

执行阶段是 Agent 生命周期里最像“干活”的部分。

它可能会：

```text
调用 API
查询数据库
操作浏览器
读写文件
发送消息
提交表单
调用另一个 Agent
```

在 MCP 场景里，Execution 通常是工具调用。

在 ACP 场景里，Execution 可能是操作 IDE、浏览器、桌面应用。

在 A2A 场景里，Execution 可能是把子任务交给另一个 Agent。

执行阶段必须区分两类动作：

```text
只读动作：查询、搜索、读取、分析
写入动作：发送、修改、删除、提交、退款
```

只读动作可以更自动化。

写入动作必须更谨慎，尤其是数字员工场景。

比如：

```text
发送客户消息
执行退款
提交审批
修改合同
删除文件
```

这些动作都应该有权限控制、确认流程和审计记录。

---

### 6. 观察反馈：Observation

Agent 执行完动作后，不能直接认为任务完成。

它必须观察外部环境反馈。

比如工具返回：

```json
{
  "success": false,
  "error_code": "permission_denied"
}
```

Agent 不能把它理解成“用户没有资格退款”。

它只能理解成：

```text
当前账号没有权限执行该操作
```

在 ACP 场景里也一样。

点击按钮返回 success，不代表页面真的提交成功。

还要观察：

```text
页面是否跳转
是否出现成功提示
是否生成任务 ID
列表中是否出现新记录
```

Observation 是避免工具幻觉的关键环节。

一个成熟 Agent 不是“做了动作就结束”，而是“根据环境反馈确认动作是否有效”。

---

### 7. 更新状态：State Update

Agent 生命周期不是一串独立调用，而是一条状态流。

每一步执行后，都应该更新 State。

状态里至少包括：

```text
当前目标
当前计划
已完成步骤
工具调用结果
失败原因
待确认事项
下一步动作
```

比如：

```json
{
  "task_id": "refund-001",
  "status": "waiting_for_confirmation",
  "completed_steps": ["query_order", "check_logistics"],
  "next_step": "confirm_refund",
  "risk": "high_write_action"
}
```

没有 State，Agent 就无法恢复任务，也无法解释自己做过什么。

这也是 Agent Runtime 和普通 Workflow 的区别之一。

Workflow 更偏静态流程。

Agent Runtime 更关注动态状态、上下文变化和执行历史。

---

### 8. 失败处理：Recovery

真实系统里，Agent 一定会失败。

失败可能来自：

```text
工具超时
权限不足
参数错误
上下文缺失
外部系统异常
用户目标变化
模型理解错误
```

成熟 Agent 生命周期必须包含失败处理。

常见方式包括：

```text
重试
换工具
降级为只读分析
请求用户确认
交给人工
记录失败状态
```

比如退款工具失败时，Agent 不应该说“已退款”。

它应该说：

```text
退款操作没有完成，原因是当前账号没有权限。我已整理好退款建议，可以提交给有权限的人处理。
```

这就是数字员工能不能进入真实业务的分水岭。

---

### 9. 结果交付：Final Response

最后一步是结果交付。

但交付不是简单说一句“完成了”。

Agent 的最终回复应该包含：

```text
做了什么
依据是什么
哪些工具执行成功
哪些没有执行
是否还有待确认事项
用户下一步需要做什么
```

尤其是涉及外部系统写入时，Agent 必须提供执行证据。

比如：

```text
已创建工单，工单号为 xxx
已生成退款建议，但未执行退款
已发送草稿，等待你确认后再发送
```

这比一句“搞定了”更可靠。

Agent 的最终回答，应该是生命周期状态的总结，而不是模型的自信表达。

---

## 工业界方案

### 1. 用 Runtime 管理生命周期

工业界不会只靠 Prompt 管 Agent。

更合理的做法是让 Runtime 管理生命周期。

Runtime 负责：

```text
任务创建
上下文加载
计划生成
工具调度
状态记录
错误恢复
权限控制
最终交付
```

模型负责推理和生成。

Runtime 负责边界和执行。

这也是 Agent 工程化的核心分工。

---

### 2. MCP 负责工具能力接入

MCP 更像 Agent 的工具连接层。

它解决的是：

```text
Agent 可以连接哪些系统
有哪些工具可以调用
工具输入输出是什么
工具能力如何被发现
```

但 MCP 本身不等于完整生命周期。

生命周期还需要 Runtime 管理：

```text
什么时候调用工具
调用失败怎么办
是否需要确认
结果如何写入 State
```

所以 MCP 是工具层，Runtime 是运行层。

---

### 3. ACP 负责环境操作反馈

ACP 更适合桌面、IDE、浏览器、本地应用这类场景。

这类场景里，Agent 不只是调用 API，而是在操作环境。

所以生命周期里必须有：

```text
操作前状态
动作执行
操作后观察
状态差异判断
```

否则 Agent 很容易以为自己完成了某个动作，但真实软件界面并没有变化。

---

### 4. A2A 负责多 Agent 协作

当任务复杂到一个 Agent 做不了，就会进入 A2A 协作。

比如一个数字员工处理合同风险：

```text
主 Agent 接收任务
合同审查 Agent 分析条款
财务 Agent 判断付款风险
法务 Agent 给出修改建议
主 Agent 汇总结果
```

这时生命周期不只是一条线，而是一棵任务树。

每个子 Agent 都有自己的生命周期。

主 Agent 需要管理：

```text
任务分发
结果收集
冲突处理
证据校验
最终汇总
```

A2A 的关键不是“多个 Agent 聊天”，而是结构化地传递任务状态和执行结果。

---

### 5. 数字员工要有审计闭环

数字员工比普通 Agent 更强调可追踪。

因为它可能代表用户进入企业系统执行真实动作。

所以它的生命周期必须可审计：

```text
谁发起了任务
Agent 如何理解任务
调用了哪些工具
参数是什么
外部系统返回什么
是否有人工确认
最终状态是什么
```

没有生命周期审计，数字员工就只是一个黑盒自动化。

有了生命周期审计，它才可能进入真实生产环境。

---

## 常见误区

### 误区一：把 Agent 生命周期理解成一次模型调用

一次模型调用只是 Agent 生命周期中的一个节点。

真正的 Agent 是持续状态管理。

它要处理上下文、工具、反馈、失败和交付。

---

### 误区二：以为 Planning 之后就按计划执行到底

真实任务里，计划经常会变。

工具返回失败，用户补充信息，外部系统状态变化，都会导致计划调整。

所以 Agent 生命周期应该支持动态规划，而不是固定脚本。

---

### 误区三：只关注工具调用，不关注状态更新

工具调用只是动作。

State 才是 Agent 能持续执行的基础。

没有 State，Agent 就不知道自己做过什么、做到哪一步、下一步该做什么。

---

### 误区四：把最终回复当成任务完成

Agent 说“完成了”，不代表真的完成了。

真正的完成需要外部系统反馈、状态确认和执行证据。

尤其是写操作场景，必须区分：

```text
已生成建议
已创建草稿
已提交系统
已被目标系统确认
```

---

## 面试加分答案

如果面试官问：

> Agent 的生命周期是什么？

可以这样回答：

```text
我理解 Agent 的生命周期，不是一次 LLM 调用，而是一个任务从创建到完成的完整运行过程。

一般包括几个阶段：首先是接收目标，把用户的自然语言需求转成任务对象；然后是构建上下文，从历史对话、文件、Memory、企业系统和 MCP 工具中收集相关信息；接着是 Planning，把目标拆成可执行步骤；然后是 Tool Selection，根据任务动态选择可用工具，并区分只读工具和高风险写操作；之后进入 Execution，调用 API、操作浏览器或通过 ACP 控制桌面环境；执行后还要 Observation，观察工具或环境反馈，确认动作是否真的成功；每一步之后都要更新 State，让 Runtime 知道任务进展、失败原因和下一步动作；如果失败，还要进入 Recovery，比如重试、降级、请求确认或转人工；最后才是 Final Response，把执行结果、证据和待处理事项交付给用户。

在工程上，生命周期应该由 Agent Runtime 管理，而不是只靠 Prompt。MCP 负责工具接入，ACP 负责环境操作反馈，A2A 负责多 Agent 协作，Runtime 负责状态、权限、执行历史和失败恢复。对于数字员工来说，还必须有完整审计链路，确保每一次工具调用、写操作和最终状态都可追踪。

所以 Agent 生命周期的核心，不是“模型想了什么”，而是“任务状态如何被可靠地推进到完成”。
```

---

## 一句话总结

Agent 的生命周期，是 Runtime 围绕一个任务持续管理目标、上下文、计划、工具调用、环境反馈、状态更新、失败恢复和最终交付的完整过程。
