# Agent 面试知识点 #001：为什么 Agent 需要任务规划（Planning）？

## 问题

主 Agent 做任务规划时，怎么保证拆解步骤合理？

或者换一种问法：

> 当用户给 Agent 一个复杂目标时，Agent 如何把目标拆成一组可执行任务？

例如用户说：

> 帮我分析小米汽车的竞争格局，并生成一份 PPT 发给老板。

这句话对人来说只是一个任务。

但对 Agent 来说，它至少包含：

1. 搜索小米汽车相关资料；
2. 搜索主要竞争对手资料；
3. 整理行业格局；
4. 形成分析结论；
5. 生成 PPT；
6. 找到老板邮箱；
7. 发送邮件。

Agent 不能直接从目标跳到结果。

它必须先做任务规划。

---

## 面试官在考什么？

这个问题表面上是在问 Prompt，实际上是在考 Agent 系统设计。

面试官真正想知道的是：

> 你是否理解 Agent 从 Goal 到 Execution 的完整链路？

一个可执行 Agent 通常要经历：

```text
Goal
↓
Plan
↓
Task
↓
Tool Call
↓
Observation
↓
Next Action
```

也就是说，Agent 不是只会回答问题，而是要把用户目标变成可执行动作。

所以 Planning 的核心不是“让模型想一想”。

而是：

> 如何把自然语言目标转化为可验证、可执行、可恢复的任务计划。

这也是 Agent 和 ChatBot 的关键区别。

ChatBot 主要回答问题。

Agent 需要管理任务。

---

## 核心知识点一：什么是 Planning？

Planning 是 Agent 根据用户目标生成执行计划的过程。

它解决的是：

> 下一步应该做什么？

例如：

```text
用户目标：生成一份竞品分析 PPT
```

一个不好的计划可能是：

```text
1. 分析竞品
2. 生成 PPT
```

这个计划太粗。

因为“分析竞品”不是一个可直接执行的动作。

一个更好的计划应该是：

```text
1. 搜索目标公司资料
2. 搜索竞品公司资料
3. 提取产品、价格、渠道、用户群信息
4. 归纳竞争格局
5. 生成 PPT 大纲
6. 生成 PPT 文件
```

更进一步，在 Agent 系统中，计划最好还要能映射到工具：

```json
{
  "steps": [
    {
      "id": "step_1",
      "goal": "搜索小米汽车资料",
      "tool": "web_search",
      "input": "小米汽车 SU7 竞争格局"
    },
    {
      "id": "step_2",
      "goal": "生成 PPT",
      "tool": "ppt_generator",
      "input": "基于竞品分析结果生成 10 页 PPT"
    }
  ]
}
```

这叫可执行计划。

可执行计划有三个特征：

1. 每一步目标清晰；
2. 每一步都能映射到工具或子 Agent；
3. 每一步的输出能被下一步使用。

---

## 核心知识点二：任务拆解为什么容易出问题？

Agent 做 Planning 时，常见问题有四类。

### 1. 漏步骤

例如：

```text
搜索资料
↓
生成 PPT
```

中间缺少：

```text
整理信息
形成结论
设计结构
```

这会导致最终产物很空。

### 2. 步骤过大

例如：

```text
分析市场
```

这不是机器可执行动作。

Agent 不知道应该搜索什么、比较什么、输出什么。

更好的拆法是：

```text
搜索市场规模
搜索主要玩家
对比产品能力
总结竞争关系
```

### 3. 工具幻觉

例如 Agent 计划里写：

```text
查询公司内部 CRM 数据
```

但系统根本没有 CRM 工具。

这就是工具幻觉。

计划看起来合理，但无法执行。

### 4. 依赖关系错误

例如：

```text
1. 生成 PPT
2. 搜索资料
```

顺序错了。

Agent 需要知道哪些任务必须先完成，哪些任务可以并行。

---

## 核心知识点三：CoT、CoT-SC 和 Planning 的关系

很多面试里会提到 CoT-SC。

CoT 是 Chain of Thought，也就是让模型先推理再回答。

在 Agent Planning 中，CoT 的作用是让模型显式思考：

```text
用户目标是什么？
需要哪些步骤？
每一步依赖什么？
最终产物是什么？
```

CoT-SC 是 Chain of Thought Self Consistency。

它的思路是：

> 不只生成一个计划，而是生成多个候选计划，再从中选择最合理的。

例如：

```text
Plan A：先搜索资料，再生成报告，再做 PPT
Plan B：先列框架，再按框架搜索资料，再生成 PPT
Plan C：先找竞品列表，再逐个分析，再形成结论
```

然后让模型或规则选择最优计划。

CoT-SC 对 Planning 有帮助，因为复杂任务往往不只有一种拆法。

但要注意：

> CoT-SC 只能提高计划质量，不能保证计划一定可执行。

因为它解决的是“想得更充分”，不是“系统一定能执行”。

真正工程化时，还需要工具约束、结构化输出、计划校验和执行反馈。

---

## 工业界方案：Plan → Validate → Execute

比较可靠的 Agent Planning 架构通常不是一步生成计划，而是分成三段：

```text
Planner
↓
Validator
↓
Executor
```

### 1. Planner：生成计划

Planner 负责根据用户目标生成任务计划。

它需要知道：

- 用户目标；
- 当前上下文；
- 可用工具；
- 任务约束；
- 期望输出。

输出最好是结构化的。

例如：

```json
{
  "goal": "生成小米汽车竞品分析 PPT",
  "steps": [
    {
      "id": "1",
      "name": "搜索小米汽车资料",
      "tool": "web_search",
      "expected_output": "小米汽车产品与市场信息"
    },
    {
      "id": "2",
      "name": "搜索竞品资料",
      "tool": "web_search",
      "expected_output": "特斯拉、极氪、问界等竞品信息"
    },
    {
      "id": "3",
      "name": "生成分析结论",
      "tool": "llm_reasoning",
      "expected_output": "竞争格局、优势、风险"
    },
    {
      "id": "4",
      "name": "生成 PPT",
      "tool": "ppt_generator",
      "expected_output": "PPT 文件"
    }
  ]
}
```

### 2. Validator：校验计划

Validator 负责检查计划是否合理。

重点检查：

- 是否漏步骤；
- 步骤是否太大；
- 是否使用了不存在的工具；
- 输入输出是否匹配；
- 依赖关系是否正确；
- 是否需要用户确认。

例如，Validator 发现：

```text
发送邮件前没有确认收件人
```

就应该要求补充：

```text
先查找或确认老板邮箱，再发送邮件。
```

### 3. Executor：执行计划

Executor 负责按计划调用工具。

执行过程中可能失败。

例如：

- 搜索结果质量差；
- 工具调用失败；
- 文件生成失败；
- 权限不足；
- 用户需要确认。

所以 Executor 还需要支持：

```text
执行
↓
观察结果
↓
失败重试
↓
必要时重新规划
```

这就是动态 Planning。

## 常见工程手段

### 1. Tool Constraint

把可用工具明确告诉 Agent。

Agent 只能在工具白名单里选。

```json
{
  "allowed_tools": [
    "web_search",
    "browser",
    "file_writer",
    "ppt_generator",
    "email_sender"
  ]
}
```

这样可以减少工具幻觉。

### 2. Plan Schema

要求计划必须按固定结构输出。

例如每一步都必须包含：

```text
step_id
objective
tool
input
expected_output
depends_on
```

这能让计划进入程序执行，而不是停留在自然语言。

### 3. Plan Validator

用规则或另一个模型检查计划。

规则适合检查硬约束：

```text
工具是否存在
字段是否完整
依赖是否成环
```

模型适合检查软约束：

```text
是否漏步骤
是否符合用户目标
是否需要补充信息
```

### 4. Reflection

执行失败后，让 Agent 反思：

```text
为什么失败？
是工具失败，还是计划不合理？
是否需要换工具？
是否需要重新拆任务？
```

Reflection 不是为了让回答更漂亮，而是为了让 Agent 可以从执行反馈中修正计划。

### 5. Human-in-the-loop

有些步骤必须让人确认。

例如：

- 发邮件；
- 删除文件；
- 提交订单；
- 修改线上配置；
- 付款；
- 对外发布内容。

好的 Planning 系统应该能识别高风险步骤，并插入人工确认。

## 常见误区

### 误区一：以为 Planning 就是写 Prompt

Prompt 很重要，但 Planning 不是 Prompt 本身。

Planning 是一个系统能力。

它包括：

```text
目标理解
任务拆解
工具选择
结构化计划
计划校验
执行反馈
重新规划
```

### 误区二：认为计划越详细越好

计划太粗不可执行。

但计划太细也会导致成本高、执行慢、容易中断。

好的计划应该是：

> 粒度刚好能被工具或子 Agent 执行。

### 误区三：只在执行前规划一次

真实任务经常变化。

搜索不到资料、工具失败、用户补充要求，都会导致计划变化。

所以 Agent 需要动态规划，而不是一次性规划。

### 误区四：没有区分计划和执行

计划是“准备做什么”。

执行是“真的调用工具做”。

两者要分开。

否则 Agent 很容易一边想一边乱调用工具，系统不可控。

---

## 面试加分答案

可以这样回答：

> 我们一般不会只依赖一次 Prompt 来做任务规划，而是采用 Plan → Validate → Execute 的结构。主 Agent 首先基于用户目标、上下文和可用工具生成结构化计划，每一步都要映射到具体 Tool 或子 Agent。然后通过 Validator 检查计划是否完整、工具是否存在、依赖关系是否正确，以及是否涉及高风险操作。执行过程中，Executor 根据 Observation 判断是否继续、重试或重新规划。对于复杂任务，可以用 CoT-SC 生成多个候选计划，再结合规则和模型选择最合理的方案。

更简洁一点：

> 合理的任务拆解不是靠模型自由发挥，而是靠工具约束、结构化计划、计划校验和执行反馈共同保证。

---

## 一句话总结

Agent Planning 的本质，是把用户的自然语言目标转化为可执行、可校验、可恢复的任务计划。

它不是单纯的 Prompt Engineering，而是 Agent Runtime 的核心能力。
