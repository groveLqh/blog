# Agent 面试知识点 #004：CoT、ToT、GoT 有什么区别？

## 摘要

CoT、ToT、GoT 这三个概念经常被放在一起讲，但它们真正的区别不在于“提示词写法不同”，而在于 Agent 如何组织自己的思考路径。

简单说：

```text
CoT：沿着一条路径一步步推理。
ToT：同时探索多条路径，并能比较、回退和选择。
GoT：把多个推理结果组织成图，可以合并、复用、反馈和迭代。
```

如果你只把它们理解成 Prompt 技巧，面试回答会比较浅。

如果你能把它们讲成 Agent Runtime 里的推理结构、搜索策略和任务状态管理，回答就会明显高一个层次。

---

## 问题

CoT、ToT、GoT 有什么区别？

在 Agent 系统里，它们分别适合什么场景？

---

## 面试官在考什么？

这个问题表面是在问三个名词：

```text
Chain of Thought
Tree of Thoughts
Graph of Thoughts
```

但面试官真正想看的是：

> 你是否理解 Agent 不是简单生成答案，而是在复杂任务中组织推理过程。

传统 ChatBot 很多时候是一次性回答。

但 Agent 面对的是任务。

任务有几个特点：

- 目标可能不清晰；
- 路径可能不唯一；
- 中间结果可能需要验证；
- 工具调用可能失败；
- 多个子任务之间可能互相依赖；
- 最终答案可能需要综合多个来源。

所以 Agent 需要的不只是“会想”，而是“会组织思考”。

CoT、ToT、GoT 本质上就是三种组织思考的结构。

---

## 核心知识点

## 1. CoT：线性推理

CoT 是 Chain of Thought，也就是“思维链”。

它的结构很简单：

```text
问题
↓
中间步骤 1
↓
中间步骤 2
↓
中间步骤 3
↓
答案
```

它适合解决路径相对明确的问题。

例如：

```text
用户问：为什么 Agent 需要任务规划？

Agent 可以按顺序推理：
1. Agent 面对的是多步骤任务；
2. 多步骤任务需要拆解；
3. 拆解之后需要决定执行顺序；
4. 所以 Agent 需要 Planning。
```

在实际系统里，CoT 更像是 Agent 的“单线程思考过程”。

它的优点是简单、成本低、容易实现。

但缺点也很明显：

一旦前面的推理方向错了，后面可能会一路错下去。

这在数字员工场景里很常见。

比如数字员工要处理一个法务合同审查任务，如果一开始把任务理解成“格式检查”，后面就可能忽略真正重要的“风险条款识别”。

所以 CoT 适合明确问题，不适合高不确定任务。

---

## 2. ToT：树状搜索

ToT 是 Tree of Thoughts，也就是“思维树”。

它不只沿着一条路径走，而是允许 Agent 同时生成多个候选思路，再进行评估和选择。

结构大概是：

```text
问题
├── 思路 A
│   ├── A1
│   └── A2
├── 思路 B
│   ├── B1
│   └── B2
└── 思路 C
    ├── C1
    └── C2
```

ToT 的关键不是“多想几个答案”，而是：

```text
生成候选 → 评估候选 → 选择路径 → 必要时回退
```

这就非常接近搜索算法。

例如 Agent 要设计一个企业数字员工方案。

它可以先生成几种路径：

```text
路径 A：从业务流程自动化切入
路径 B：从企业系统连接层切入
路径 C：从 Agent Runtime 能力切入
路径 D：从权限、审计、风控切入
```

然后评估：

```text
哪个方案最快落地？
哪个方案复用价值最高？
哪个方案最容易形成平台能力？
哪个方案风险最大？
```

最后选择一个主路径，或者组合多个路径。

ToT 在 Agent 里特别适合这些场景：

- 复杂规划；
- 代码修复；
- 多方案设计；
- 需求分析；
- 任务拆解；
- 失败后重新规划。

它比 CoT 更强，但也更贵。

因为它要生成、评估、裁剪多个分支。

所以工业界通常不会所有任务都用 ToT，而是在高价值、高不确定、高失败成本的任务里使用。

---

## 3. GoT：图状推理

GoT 是 Graph of Thoughts，也就是“思维图”。

它比 ToT 更进一步。

ToT 里的路径大多是树状的，一个节点通常只有一个上游路径。

但真实任务并不总是树。

很多任务更像图。

```text
资料 A ─┐
        ├── 中间结论 X ─┐
资料 B ─┘              ├── 最终方案
资料 C ───── 中间结论 Y ┘
        ↑              ↓
        └── 反馈修正 ←──┘
```

GoT 的价值在于：

- 不同思路可以合并；
- 中间结论可以复用；
- 某个节点可以被多个任务依赖；
- 结果可以被反馈修正；
- 多个 Agent 的输出可以汇总成一个推理网络。

这在企业 Agent 里非常重要。

比如一个法务数字员工要完成“合同风险评审”。

它可能需要多个子 Agent：

```text
条款识别 Agent
风险分类 Agent
历史案例检索 Agent
业务上下文分析 Agent
输出润色 Agent
```

这些 Agent 的输出不是简单串行关系。

历史案例检索会影响风险分类。

业务上下文会影响风险等级。

风险分类又会反过来要求重新检索相关条款。

这就是图状依赖。

如果只用 CoT，会太单线。

如果只用 ToT，会偏搜索树。

而 GoT 更适合描述这种多源输入、多轮反馈、多节点依赖的复杂系统。

---

## 工业界方案

在真实工业系统里，CoT、ToT、GoT 很少以纯论文形式直接出现。

它们通常会被 Runtime、Workflow、Multi-Agent 框架吸收掉。

### 1. CoT 通常变成单 Agent 的推理轨迹

例如一个 Agent 在执行任务时，会保留：

```text
当前目标
当前步骤
已观察信息
下一步动作
最终回答
```

这就是一种工程化的 CoT。

在 ReAct 结构里，Thought 本身就带有 CoT 的影子。

```text
Thought → Action → Observation
```

### 2. ToT 通常变成规划器和评估器

工业界不会让模型无限发散。

通常会加上约束：

```text
最多生成 3 个方案
每个方案评估成本、风险、成功率
选择评分最高的方案执行
失败后最多重试 2 次
```

这就把 ToT 变成了可控的搜索策略。

例如在 AI Coding Agent 里：

```text
方案 A：直接改函数
方案 B：补测试后再改
方案 C：重构模块边界
```

Runtime 会选择风险更低、验证更充分的方案。

### 3. GoT 通常变成 Agent Runtime 的状态图

GoT 在工程里最像什么？

我觉得最像：

```text
State Graph
Task Graph
Knowledge Graph
Execution DAG
```

也就是说，GoT 不只是提示词，而是 Agent Runtime 的状态组织方式。

例如：

- LangGraph 这类框架用图来描述 Agent 状态流转；
- MCP 解决 Agent 和外部工具、数据源的连接；
- A2A 解决 Agent 与 Agent 之间的协作通信；
- ACP 更偏向 Agent 和应用、编辑器、运行环境之间的交互协议；
- 数字员工则需要把这些能力串起来，形成“输入 → 推理 → 工具执行 → 状态更新 → 输出”的闭环。

这时候 GoT 的思想就非常有价值。

因为企业任务不是一次问答，而是一张持续变化的任务网络。

## 常见误区

### 误区一：把 CoT、ToT、GoT 都理解成 Prompt 模板

这是最常见的问题。

它们当然可以通过 Prompt 实现一部分能力。

但在 Agent 系统里，更重要的是它们对应的推理结构：

```text
CoT：链
ToT：树
GoT：图
```

真正有工程价值的是结构，而不是那几句提示词。

### 误区二：认为越复杂越好

不是所有任务都需要 ToT 或 GoT。

例如简单问答、简单分类、简单信息抽取，用 CoT 甚至不用 CoT 都可以。

如果动不动就上 ToT、GoT，成本会变高，延迟会变长，系统也更难调试。

### 误区三：把 ToT 等同于多 Agent

ToT 是多路径搜索，不一定是多 Agent。

一个 Agent 也可以做 ToT。

多 Agent 是多个角色或多个执行单元协作。

两者可以结合，但不是一回事。

### 误区四：忽略 Runtime

面试里只讲 Prompt，很容易显得停留在 Demo 层。

真正的 Agent 系统需要 Runtime 负责：

- 记录状态；
- 管理分支；
- 控制回退；
- 调用工具；
- 合并结果；
- 做权限控制；
- 做成本控制；
- 做可观测性。

没有 Runtime，ToT 和 GoT 很难稳定落地。

---

## 面试加分答案

如果面试官问：

> CoT、ToT、GoT 有什么区别？

可以这样回答：

```text
我不会把它们只看成 Prompt 技巧，而会把它们看成 Agent 推理结构的三种复杂度。

CoT 是链式推理，适合路径明确的多步骤问题，成本低，但容易一条路走到黑。

ToT 是树状搜索，适合需要探索多个方案的任务，例如规划、代码修复、复杂需求分析。它允许生成多个候选路径，并通过评估、选择和回退提高成功率，但成本更高。

GoT 是图状推理，适合多源信息、多 Agent 协作、多轮反馈的复杂任务。它允许中间结论复用、合并和反馈修正，更接近企业级 Agent Runtime 中的任务图或状态图。

在工业界，我会根据任务复杂度选择：简单任务用 CoT；高不确定任务用 ToT；跨系统、跨 Agent、强依赖关系的企业任务用 GoT 思想设计 Runtime。
```

这个回答的加分点在于：

你没有停留在名词解释，而是把它们和系统设计联系起来了。

---

## 一句话总结

CoT、ToT、GoT 的区别，本质上是 Agent 推理结构从“一条链”，升级为“多棵树”，再升级为“一张图”。

真正的 Agent 工程，不是让模型多想几步，而是让 Runtime 能管理这些思考路径。
