# Agent 面试知识点 #008：Agent 如何避免工具幻觉？

## 摘要

Agent 的工具幻觉，指的是模型在调用工具时出现不真实、不合理或不可执行的行为。

比如：

```text
调用一个不存在的工具
编造工具参数
误以为工具已经执行成功
把工具返回结果理解错
在没有权限的情况下尝试写操作
把自然语言推测当成真实系统状态
```

工具幻觉不是单纯的 Prompt 问题。

它本质上是 Agent Runtime 没有把工具能力、工具边界、执行结果和系统状态管理清楚。

一个成熟的 Agent 系统，不能只让模型“知道有哪些工具”，还要让 Runtime 负责工具注册、参数校验、权限控制、结果验证、状态更新和失败处理。

---

## 问题

Agent 如何避免工具幻觉？

在 MCP、ACP、A2A 和数字员工场景里，如何防止 Agent 乱调用工具、误解工具结果，或者假装完成任务？

---

## 面试官在考什么？

这个问题表面是在问 Tool Calling，实际上是在考你是否理解 Agent 工程里的一个核心原则：

> 工具不是模型能力的一部分，而是 Runtime 管理的外部能力。

普通 ChatBot 可以只生成文本。

但 Agent 会调用工具、操作浏览器、读写文件、访问企业系统，甚至代表用户执行真实动作。

这时问题就变复杂了。

比如一个数字员工要处理客户退款：

```text
读取客户消息
查询订单
判断是否符合退款条件
调用退款系统
生成回复
通知用户
```

如果 Agent 误以为自己已经查过订单，或者误把“查询失败”理解成“订单不存在”，就可能做出错误决策。

更严重的是，如果它调用写操作工具，比如：

```text
refund_order
send_message
update_contract
submit_approval
```

工具幻觉就不只是回答错误，而可能变成业务事故。

所以面试官想听到的不是：

```text
把工具描述写清楚一点
```

而是：

```text
工具幻觉要靠 Runtime 机制治理，包括工具注册、Schema 校验、权限边界、执行状态、结果校验和人工确认。
```

---

## 核心知识点

### 1. 工具幻觉的第一类：编造工具

模型可能会凭语义想象一个工具。

比如系统只有：

```text
query_order
create_ticket
send_message
```

但模型可能会调用：

```text
refund_user
check_payment_risk
auto_solve_complaint
```

因为这些名字“看起来合理”。

避免方式不是只靠提醒模型，而是 Runtime 必须做工具白名单。

```text
只有注册过的工具才能调用
未注册工具直接拒绝
拒绝原因写入 State
要求 Agent 重新规划
```

在 MCP 场景里尤其如此。

MCP server 暴露了哪些工具，Agent 只能使用哪些工具，不能凭空创造工具。

---

### 2. 工具幻觉的第二类：编造参数

即使工具存在，参数也可能是错的。

比如工具要求：

```json
{
  "order_id": "string",
  "reason": "string"
}
```

但 Agent 可能传：

```json
{
  "user_name": "张三",
  "refund": true
}
```

或者更隐蔽一点：

```json
{
  "order_id": "我认为是 12345"
}
```

这里的问题是，模型把推测当成事实。

所以工具调用必须依赖 Schema 校验：

```text
参数类型是否正确
必填字段是否存在
枚举值是否合法
金额、时间、ID 是否符合格式
高风险参数是否来自可信上下文
```

参数校验失败时，不应该直接执行，而应该进入重新规划或请求用户补充信息。

---

### 3. 工具幻觉的第三类：误解工具结果

工具返回结果不等于 Agent 理解正确。

比如工具返回：

```text
status: timeout
```

Agent 却理解成：

```text
订单不存在
```

工具返回：

```text
permission_denied
```

Agent 却理解成：

```text
用户没有退款资格
```

这就是结果解释幻觉。

所以工具结果需要结构化，而不是只返回一段自然语言。

更好的返回格式是：

```json
{
  "success": false,
  "error_code": "permission_denied",
  "message": "当前账号无权限查询该订单",
  "next_action": "request_permission"
}
```

Runtime 要把工具结果写入 State，让 Agent 基于真实状态继续决策，而不是基于想象继续生成。

---

### 4. 工具幻觉的第四类：假装完成

这是 Agent 系统里最危险的一类。

比如用户让 Agent：

```text
帮我把这份合同发给法务负责人
```

Agent 最后回复：

```text
已发送。
```

但实际上工具调用失败了，或者根本没有调用发送工具。

这种问题在数字员工里非常严重。

因为用户会默认它真的执行了。

解决方式是：最终回答必须绑定执行证据。

```text
是否调用了工具
工具调用是否成功
返回了什么任务 ID
目标系统是否确认
是否有可追踪日志
```

也就是说，Agent 不能只说“完成了”，必须能证明“完成了”。

---

## 工业界方案

### 1. 工具注册由 Runtime 管理

工具不能只是写在 Prompt 里。

成熟系统会有统一工具注册中心：

```text
工具名称
工具描述
输入 Schema
输出 Schema
权限级别
是否有副作用
是否需要用户确认
调用成本
失败重试策略
```

模型只能从 Runtime 暴露的工具集合里选择。

这样可以防止 Agent 编造不存在的工具。

---

### 2. MCP 工具要区分读操作和写操作

MCP 很适合让 Agent 连接外部系统。

但所有工具不能一视同仁。

至少要分成：

```text
只读工具：查询、搜索、读取文件
低风险写工具：创建草稿、生成记录
高风险写工具：发送消息、提交审批、转账、删除、退款
```

只读工具可以自动调用。

高风险写工具必须进入确认流程。

比如：

```text
Agent 生成操作计划
Runtime 展示影响范围
用户确认
工具执行
记录结果
```

这能避免 Agent 因幻觉直接执行不可逆动作。

---

### 3. ACP 场景要用环境反馈校验动作结果

ACP 偏向操作桌面、IDE、浏览器和本地应用。

这类工具幻觉经常表现为：

```text
Agent 以为点到了按钮
Agent 以为页面跳转成功
Agent 以为文件保存成功
Agent 以为命令执行成功
```

但真实环境可能没有变化。

所以 ACP 操作必须有反馈校验：

```text
操作前状态
执行动作
操作后状态
状态差异
是否符合预期
```

比如点击“提交”后，不能只看工具是否返回 success，还要看页面是否出现：

```text
提交成功
任务 ID
下一步状态
列表中出现新记录
```

Agent 操作真实软件时，必须用环境状态验证，而不是相信自己“已经做了”。

---

### 4. A2A 协作要传递结构化结果

多 Agent 协作里，工具幻觉会被放大。

一个子 Agent 如果返回：

```text
我已经分析完了，风险不大。
```

主 Agent 很难判断这个结论是否可靠。

更好的 A2A 返回应该是：

```json
{
  "status": "completed",
  "confidence": 0.82,
  "evidence": ["条款 3.2", "条款 5.1"],
  "risks": [
    {
      "level": "medium",
      "reason": "违约责任不明确"
    }
  ],
  "requires_human_review": true
}
```

主 Agent 不能盲信子 Agent 的自然语言结论。

它需要结构化状态、证据、置信度和是否需要人工复核。

---

### 5. 数字员工必须有执行审计

数字员工连接企业环境后，工具幻觉不再只是技术问题，而是治理问题。

所以需要完整审计链路：

```text
用户目标是什么
Agent 计划是什么
调用了哪些工具
参数是什么
工具返回什么
是否发生写操作
谁确认的
最终结果是什么
```

这套审计链路可以用于：

```text
问题排查
安全合规
责任边界
效果评估
后续训练
```

没有审计的数字员工，本质上是不可控的自动化。

## 常见误区

### 误区一：以为 Prompt 写清楚就够了

Prompt 很重要，但不够。

模型仍然可能误解、遗漏、过度推断。

工具幻觉必须靠 Runtime 兜底。

---

### 误区二：把所有工具都开放给 Agent

工具越多，Agent 选择错误的概率越高。

应该按任务上下文动态暴露工具，而不是一次性把所有工具都给模型。

比如处理订单问题时，只暴露订单、售后、消息相关工具。

不要把合同、财务、审批工具也一起暴露。

---

### 误区三：只校验输入，不校验输出

很多系统只做参数校验，却不校验工具结果。

但工具调用成功，不代表业务成功。

比如接口返回 200，不等于审批提交成功。

Runtime 需要校验业务状态是否真的变化。

---

### 误区四：让 Agent 自己判断高风险动作

高风险动作不能完全交给模型判断。

比如：

```text
删除文件
发送客户通知
提交审批
执行退款
修改合同
```

这些都应该由系统规则判断是否需要人工确认。

模型可以提出建议，但 Runtime 决定能不能执行。

---

## 面试加分答案

如果面试官问：

> Agent 如何避免工具幻觉？

可以这样回答：

```text
我理解工具幻觉不是单纯的模型问题，而是 Agent Runtime 的治理问题。

它主要包括几类：第一，模型编造不存在的工具；第二，调用真实工具但编造参数；第三，误解工具返回结果；第四，在没有真实执行成功的情况下假装任务完成。

解决这个问题不能只靠 Prompt，而要从 Runtime 层做控制。

首先，工具必须通过注册中心管理，只有白名单工具可以被调用，并且每个工具都有输入 Schema、输出 Schema、权限级别和副作用标记。

其次，MCP 工具要区分读操作和写操作。只读工具可以自动执行，高风险写操作必须有人确认。

第三，ACP 场景要基于环境反馈校验动作结果，不能因为点击工具返回成功，就认为真实页面已经完成提交。

第四，A2A 多 Agent 协作中，子 Agent 不能只返回自然语言结论，而要返回结构化状态、证据、置信度和是否需要人工复核。

最后，数字员工必须有完整审计链路，包括目标、计划、工具调用、参数、结果、确认人和最终状态。

所以避免工具幻觉的核心，不是让模型“更听话”，而是让 Runtime 对工具能力、调用边界、执行结果和状态变化形成闭环控制。
```

这个回答的加分点在于：

你没有停留在“优化 Prompt”或“减少工具数量”，而是把工具幻觉放到了 Agent Runtime、MCP 工具治理、ACP 环境反馈、A2A 协作协议和数字员工审计体系里。

---

## 一句话总结

Agent 避免工具幻觉的关键，不是相信模型会正确使用工具，而是由 Runtime 明确工具边界、校验调用参数、验证执行结果，并在高风险场景中引入权限控制和人工确认。
