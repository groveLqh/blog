# Agent 面试知识点 #012：RAG 和 Memory 的区别是什么？

## 摘要

很多人会把 RAG 和 Memory 混在一起。

因为它们看起来都像是在做一件事：

```text
把模型不知道的信息找出来
再放进上下文里
```

但这只是表面相似。

RAG 更像“查资料”。

Memory 更像“积累经验”。

RAG 解决的是：

```text
当前任务需要哪些外部知识？
这些知识在哪里？
如何把相关片段检索出来？
```

Memory 解决的是：

```text
Agent 在过去任务中学到了什么？
用户有哪些稳定偏好？
项目有哪些长期规则？
哪些执行经验以后还会复用？
```

所以，RAG 关注知识检索，Memory 关注长期沉淀。

它们都可能进入 Context，但它们不是同一个东西。

---

## 问题

RAG 和 Memory 的区别是什么？

为什么不能把 Memory 简单理解成一个向量数据库？

在 Agent 系统里，什么时候该用 RAG，什么时候该用 Memory？

---

## 面试官在考什么？

这个问题表面上是在问两个概念的区别。

但面试官真正想考的是：

> 你是否理解 Agent 的信息来源、长期状态和上下文管理不是一回事。

很多候选人会这样回答：

```text
RAG 是检索知识，Memory 也是检索历史，所以差不多。
```

这个回答太浅。

真正的差异在于：

```text
RAG 通常面向外部知识
Memory 通常面向 Agent 自身经验
RAG 多数是读多写少
Memory 是持续读写更新
RAG 更强调召回和相关性
Memory 更强调生命周期、可信度、权限和沉淀价值
```

比如 AI Coding Agent 需要理解某个框架 API，这更像 RAG。

它需要记住“这个项目使用 pnpm、测试命令是 pnpm test、团队要求 review 时优先看兼容性”，这更像 Memory。

这两个信息最后都可能被塞进 Context Window，但来源、更新方式和治理方式不同。

---

## 核心知识点

### 1. RAG 是知识检索，Memory 是经验沉淀

RAG 的核心是 Retrieval-Augmented Generation。

它的基本流程是：

```text
用户问题
  ↓
检索相关文档、代码、网页、知识库片段
  ↓
把结果放入上下文
  ↓
模型基于检索内容回答或执行
```

它适合解决模型参数里没有、当前上下文里也没有的信息。

比如：

```text
公司内部 API 文档
某个项目 README
历史 issue
产品需求文档
MCP 工具说明
第三方 SDK 用法
```

Memory 的核心不是“找资料”，而是“沉淀 Agent 运行中形成的有效信息”。

比如：

```text
用户喜欢 Markdown 结构化输出
某个项目不要随便修改公共组件
这个仓库的测试命令经常要加 --runInBand
某个浏览器后台提交后需要刷新列表确认结果
某类代码 review 要重点检查隐藏副作用
```

这些内容未必来自外部知识库，而是来自 Agent 和用户、工具、项目环境长期互动后的总结。

---

### 2. RAG 通常是外部知识，Memory 通常是内部状态

RAG 的信息来源通常是外部的。

比如：

```text
文档库
网页
代码仓库
数据库
PDF
知识库
搜索引擎
```

Memory 的信息来源通常是 Agent 的运行过程。

比如：

```text
用户多次表达的偏好
任务执行中的成功经验
工具调用失败后的修正方式
项目长期约束
跨会话保留的工作状态
```

在 Agent Runtime 里，这两类信息应该分开管理。

RAG 更像一个“知识访问层”。

Memory 更像一个“长期状态层”。

如果把它们混在一起，系统会很快变得不可控：不知道哪些是客观知识，哪些是用户偏好，哪些是临时状态，哪些是过期经验。

---

### 3. RAG 多数读多写少，Memory 必须持续更新

RAG 的典型使用方式是读。

文档先被采集、切块、索引，然后在任务中检索。

当然，知识库也会更新，但它通常不是由 Agent 每轮对话自动写入。

Memory 不一样。

Memory 的关键能力正是持续更新。

Agent 做完任务后，需要判断：

```text
这次是否产生了可复用经验？
用户是否表达了稳定偏好？
项目规则是否被确认？
某个工具失败模式是否值得记录？
旧记忆是否需要覆盖或删除？
```

所以 Memory 不是一次性索引，而是一个长期维护过程。

这也是为什么 Memory 更依赖 Runtime 的治理能力，而不是只依赖向量检索。

---

### 4. RAG 解决“知道什么”，Memory 解决“以后怎么做”

RAG 更常回答：

```text
这个 API 怎么用？
这份文档里怎么规定？
这个函数在哪里定义？
这个需求文档说了什么？
```

Memory 更常回答：

```text
这个用户希望我怎么写？
这个项目通常怎么测试？
这个任务上次为什么失败？
这个工具调用有什么坑？
下次遇到类似问题应该注意什么？
```

所以 RAG 更偏知识。

Memory 更偏行为。

一个好的 Agent 不只是知道更多资料，还应该越来越理解自己所在的工作环境。

这正是 Memory 的价值。

---

### 5. RAG 可以作为 Memory 的实现方式之一，但不是 Memory 本身

很多系统会把 Memory 存进向量数据库，然后通过相似度检索取回来。

这没有问题。

但不能因此说：

```text
Memory = 向量数据库
Memory = RAG
```

向量数据库只是存储和检索手段。

Memory 是一套机制。

它至少包含：

```text
写入判断：什么值得记
存储结构：记成文本、结构化字段还是事件日志
读取策略：当前任务需要哪些记忆
冲突处理：新旧记忆矛盾时怎么办
过期机制：哪些记忆不再有效
权限控制：哪些记忆可以被读写
可解释性：为什么用了这条记忆
```

RAG 主要解决其中的“检索”部分。

Memory 还要解决“生成、更新、治理和遗忘”。

---

## 工业界方案

### 1. 把 RAG 和 Memory 放在不同层里

成熟 Agent 系统通常不会把所有信息都扔进一个向量库。

更常见的架构是：

```text
用户输入
  ↓
Context Manager
  ↓
读取相关 Memory
  ↓
调用 RAG 检索外部知识
  ↓
组合当前任务上下文
  ↓
模型规划与执行
  ↓
工具调用 / 浏览器操作 / 代码修改
  ↓
Runtime 判断是否写入 Memory
```

这里的关键是分层。

RAG 是知识检索能力。

Memory 是长期状态能力。

Context Manager 决定当前任务要把哪些信息放进上下文。

Runtime 决定哪些信息可以长期保存。

---

### 2. AI Coding 场景里的典型分工

在 AI Coding 里，RAG 可以用来检索：

```text
代码仓库相关文件
接口定义
框架文档
历史 PR
错误日志
测试说明
```

Memory 可以用来保存：

```text
项目使用 pnpm
常用测试命令是 pnpm test
提交前要检查 lint
用户偏好先解释风险再给方案
某个目录是历史包袱，改动要谨慎
某类问题上次通过修改配置解决
```

RAG 让 Agent 找到当前问题相关资料。

Memory 让 Agent 下次不用重新摸索项目习惯。

两者结合后，Agent 才更像一个熟悉项目的工程助手。

---

### 3. Browser Use 和 Computer Use 场景里的典型分工

在 Browser Use 里，RAG 可以用来检索：

```text
系统操作手册
后台字段说明
业务规则文档
常见问题说明
```

Memory 可以用来保存：

```text
这个页面提交后要等列表刷新
某个按钮文案经常变化，但位置稳定
某个流程需要先切换组织再操作
某类表单字段用户习惯填写某种格式
```

在 Computer Use 里也是类似。

RAG 告诉 Agent “文档里说应该怎么做”。

Memory 告诉 Agent “以前实际操作时有哪些经验”。

前者偏说明书，后者偏工作习惯。

---

### 4. MCP、A2A、ACP 场景里的 Memory 价值

MCP 让 Agent 可以连接更多工具和数据源。

这时 RAG 可以帮助 Agent 理解工具文档、Schema 和使用说明。

Memory 则可以沉淀：

```text
哪些 MCP 工具更稳定
某个工具失败后如何重试
某类任务优先调用哪个工具
用户不希望自动执行哪些高风险操作
```

A2A 场景里，RAG 可以检索共享知识库。

Memory 更适合保存 Agent 之间的交接状态、历史结论和协作偏好。

ACP 场景里，RAG 可以检索协议文档和 IDE 能力说明。

Memory 则可以记住具体项目、具体用户、具体工作区里的长期使用经验。

---

### 5. Memory 写入通常需要更严格的治理

RAG 检索错了，通常是召回质量问题。

Memory 写错了，影响会更长期。

比如 Agent 错误记住：

```text
这个项目不需要跑测试
用户喜欢极简回答
某个工具失败就不要再用
```

这会污染之后很多任务。

所以 Memory 写入需要更谨慎。

工业界通常会做：

```text
高置信度才自动写入
重要偏好需要用户确认
敏感信息默认不写入
旧记忆需要可覆盖
用户可以查看和删除记忆
执行经验需要附带来源和时间
```

这也是 Memory 和普通 RAG 最大的工程差异之一。

---

## 常见误区

### 误区一：把 Memory 当成 RAG 的别名

RAG 是检索增强生成。

Memory 是长期经验和状态管理。

它们可以结合，但不是一回事。

---

### 误区二：认为有向量数据库就有 Memory

向量数据库只能解决相似内容检索。

Memory 还需要写入、更新、删除、冲突处理、权限和可解释性。

---

### 误区三：把所有聊天历史都当 Memory

聊天历史只是原始材料。

真正有用的 Memory 应该是被筛选和压缩后的长期信息。

---

### 误区四：RAG 和 Memory 都直接塞进上下文就行

上下文窗口有限，而且容易被噪音污染。

真正的系统需要 Context Manager 决定哪些信息进入当前任务。

---

### 误区五：忽视 Memory 的过期问题

项目规则会变，用户偏好会变，工具行为也会变。

Memory 不是越多越好，而是要保持新鲜、准确和可控。

---

## 面试加分答案

如果面试官问：

> RAG 和 Memory 的区别是什么？

可以这样回答：

```text
我理解 RAG 和 Memory 都可能给 Agent 提供额外上下文，但它们解决的问题不同。

RAG 更像外部知识检索，解决的是当前任务需要哪些文档、代码、网页或知识库信息。它通常读多写少，重点是召回、相关性和引用依据。

Memory 更像 Agent 的长期状态和经验沉淀，解决的是跨任务、跨会话之后 Agent 应该记住什么。比如用户偏好、项目规则、工具失败经验、历史任务结论和执行习惯。它必须持续更新，也必须考虑写入标准、过期、冲突、权限和可删除。

在工程实现上，RAG 可以是 Memory 的一种检索手段，但 Memory 不等于向量数据库。一个完整的 Memory 系统不仅要能查，还要能判断什么值得记、什么时候更新、什么时候遗忘，以及当前任务是否应该使用某条记忆。

所以我会把 RAG 放在知识访问层，把 Memory 放在长期状态层，再由 Context Manager 决定当前任务要组合哪些信息进入模型上下文。这样 Agent 才能既查得到资料，又能逐渐熟悉用户、项目和工具环境。
```

---

## 一句话总结

RAG 是让 Agent 在当前任务中查到相关知识，Memory 是让 Agent 在长期运行中沉淀经验、偏好和状态；RAG 更像查资料，Memory 更像长经验。
